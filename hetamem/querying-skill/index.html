    Querying Knowledge and Memory — Agent Skill

    This page describes the orchestration logic bundled with Heta as a reusable agent skill (skills/querying-knowledge-and-memory/SKILL.md). The skill is loaded automatically by Claude Code from the skills/ directory. For other clients, copy the skill content into the agent's system prompt.

    Three Systems at a Glance

    System Built by Role
    HetaDB Humans (file upload) Multimodal document knowledge bases
    MemoryKB Agent (text insert) Agent's own accumulating knowledge graph
    MemoryVG Agent (conversation) Fast personal memory and cross-session cache

    All three systems share a single base URL: http://<host>:8000

    Three-Step Orchestration

    When the agent needs to answer a knowledge question, it always follows this order:

    Step 1 — Search MemoryVG  (fast personal recall)
  → Answer is clear and specific  →  return it, done
  → Vague or absent               →  continue to Step 2

Step 2 — Query HetaDB  (human knowledge base)
  → Synthesised answer + citations returned

Step 3 — Agent decides what to store:
  → Quick recall next time?  →  MemoryVG add
  → Worth accumulating?      →  MemoryKB insert
  → Ephemeral / trivial?     →  skip

    This pattern keeps first-time lookups accurate (HetaDB) and repeat lookups fast (MemoryVG). The score threshold in Step 1 is 0.85 — only treat a MemoryVG hit as a definitive answer if score >= 0.85.

    API Quick Reference

    HetaDB — Document Knowledge Bases

    GET  /api/v1/hetadb/files/knowledge-bases
    Returns data[].name — use as kb_id. Skip KBs with status: "deleting".

    POST /api/v1/hetadb/chat
{ "query": "...", "kb_id": "...", "user_id": "agent", "query_mode": "naive" }
    user_id is always "agent". Read the response field from the result.

    MemoryVG — Personal Memory Cache

    POST /api/v1/hetamem/vg/search
{ "query": "...", "agent_id": "agent" }
    Read results[].memory. Only use results with score >= 0.85.

    POST /api/v1/hetamem/vg/add
{ "messages": [{"role": "assistant", "content": "..."}], "agent_id": "agent" }
    The LLM extracts facts from the messages. Memories are available immediately.

    MemoryKB — Agent Knowledge Graph

    POST /api/v1/hetamem/kb/insert          (multipart/form-data)
query=<text to add to knowledge graph>


    POST /api/v1/hetamem/kb/query
{ "query": "...", "mode": "hybrid" }
    Read final_answer from the result.

    When NOT to Use Each System

    Don't use For
    HetaDB Storing agent observations — it has no insert API
    MemoryKB Quick recall — the ~200 s async delay makes it unsuitable as a cache
    MemoryVG Deep multi-document reasoning — use HetaDB with rerank or multihop instead

    End-to-End Example

    import httpx

BASE = "http://localhost:8000"

def ask(question: str, kb_id: str, agent_id: str = "agent") -> str:
    # Step 1: fast personal memory check
    vg = httpx.post(f"{BASE}/api/v1/hetamem/vg/search",
                    json={"query": question, "agent_id": agent_id}).json()
    hits = [r["memory"] for r in vg.get("results", []) if r["score"] > 0.85]
    if hits:
        return f"[memory] {hits[0]}"

    # Step 2: query the document knowledge base
    resp = httpx.post(f"{BASE}/api/v1/hetadb/chat", json={
        "query":      question,
        "kb_id":      kb_id,
        "user_id":    agent_id,
        "query_mode": "rerank",
    }).json()
    answer = resp.get("response", "")

    # Step 3: cache for fast recall next time
    if answer:
        httpx.post(f"{BASE}/api/v1/hetamem/vg/add", json={
            "messages":  [{"role": "assistant", "content": answer}],
            "agent_id":  agent_id,
        })

    return answer

    Tip

    The first call hits HetaDB and caches the result in MemoryVG. The same question asked again is served from MemoryVG in ~100 ms — no LLM retrieval required.
